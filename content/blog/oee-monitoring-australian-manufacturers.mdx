OEE is one of the most used metrics in manufacturing and one of the most misused. Australian food and beverage, packaging, and FMCG plants regularly report OEE figures that look plausible but do not reflect actual production performance. The problem is almost always with data quality, not calculation method.

This guide covers how to build an OEE monitoring system on an Australian production site that captures accurate, consistent data and turns it into decisions that actually improve throughput.

## What OEE measures and what it does not

Overall Equipment Effectiveness measures three components:

- **Availability**: Planned production time minus unplanned downtime, expressed as a percentage of planned time
- **Performance**: Actual production speed as a percentage of rated or ideal speed
- **Quality**: Good product output as a percentage of total product started

OEE is the product of these three figures. A line running at 90% availability, 95% performance, and 98% quality produces an OEE of approximately 84%.

What OEE does not measure: changeover time (usually excluded from planned production time), scheduled maintenance, or planned downtime. These are real capacity constraints but are accounted for separately in overall plant efficiency frameworks such as TEEP.

Understanding these boundaries matters because poorly defined planned time is the most common source of misleading OEE figures in Australian plants.

## Why most OEE data is unreliable

Before investing in OEE software, it is worth understanding why existing data is unreliable. These are the most common failure modes.

**Planned production time is defined too loosely.** If changeovers, scheduled cleaning, and planned maintenance are inconsistently coded as planned or unplanned, availability figures vary based on who is entering data rather than actual production performance.

**Downtime reasons are too generic.** If every stoppage is coded as "mechanical fault" or "awaiting material," the downtime Pareto is useless for directing maintenance effort. Reason code design is one of the highest-leverage investments in an OEE implementation.

**Speed losses are not captured.** Many plants record downtime events but ignore chronic minor stoppages and speed reductions. A line running at 70% of rated speed is often invisible in OEE data that only counts complete stoppages.

**Data is entered retrospectively.** When operators enter downtime data at the end of a shift, accuracy degrades. Events are missed, durations are estimated, and reasons are approximated. Real-time data capture directly from PLC and SCADA events is significantly more reliable.

**No baseline was established before improvement started.** Without a credible baseline, it is impossible to measure whether improvement efforts have worked. Many OEE programs lack baseline data because measurement only started after improvement initiatives began.

## How to capture OEE data automatically from PLC and SCADA systems

Modern control systems can capture most of the data needed for OEE calculation without operator input. The architecture depends on what infrastructure exists.

### From PLC data alone

PLCs track machine running state, speed, product counts, and fault conditions in real time. The key is extracting this data with context, not just raw tag values.

**Run/stop signals** can be used to calculate machine availability, but they need supplementary logic to distinguish genuine faults from operator-initiated stops, planned changeovers, and speed tests. This context is typically available in the PLC program but needs to be explicitly tagged for historian collection.

**Cycle counters and production counts** provide the raw input for performance rate calculations. Tying these to the current product recipe gives actual versus ideal speed comparison.

**Fault code registers** provide downtime reason data from the machine controller level. Where fault codes are well-maintained and descriptive, this eliminates manual reason coding for machine-caused stoppages.

### From SCADA and historian data

Historians that record machine state transitions can reconstruct downtime event logs automatically. Ignition Historian, FactoryTalk Historian, and SQL-based approaches all support this pattern.

The calculation engine for OEE can sit in the historian, in a reporting layer, or in a dedicated OEE application. The choice depends on existing infrastructure and the reporting flexibility required.

### Where operator input remains necessary

Even in well-instrumented plants, some downtime categories require operator classification:

- Material supply delays (where no sensor confirms material availability)
- Process quality holds (where in-line checks are unavailable)
- Minor stoppages that are cleared without fault acknowledgement
- Changeover reason codes where product scheduling context is not automated

The practical approach is to automate collection of everything the PLC and SCADA can provide, then focus operator input on the categories that genuinely require human context.

## Designing downtime reason code structures

Reason code design is underestimated in almost every OEE implementation. The goal is a structure that is specific enough to direct maintenance action but not so granular that operators cannot apply it consistently under shift pressure.

A practical three-tier structure works well on most Australian production lines:

| Tier 1 (broad category) | Tier 2 (specific category) | Example Tier 3 (optional detail) |
| --- | --- | --- |
| Mechanical | Conveyor | Belt slip, chain failure, jam |
| Mechanical | Filling equipment | Valve failure, pump fault, nozzle block |
| Electrical | Drive fault | Drive trip, motor overload, encoder fault |
| Process | Material supply | Ingredient short, material flow, CIP pending |
| Process | Quality hold | Checkweigher reject, metal detection, batch release |
| Operator | Operator-initiated | Speed reduction, manual mode, training |

The key rule: codes should match how maintenance and production teams describe problems in practice. Codes developed in isolation from the people who will use them are invariably abandoned or used inconsistently.

## OEE reporting that drives decisions

Data without decisions is wasted effort. OEE reporting should be designed to answer the specific questions production managers, team leaders, and maintenance planners ask every day.

**Daily shift summary.** Availability, performance, quality, and OEE by line and shift. Top three downtime events with duration and reason code. This is the primary tool for shift handover and daily production meetings.

**Weekly downtime Pareto.** Ranked downtime events by total minutes lost over the week. This is the primary input for maintenance prioritisation and continuous improvement planning.

**Changeover performance.** Changeover start, end, and duration by product transition. This is often the highest-leverage improvement opportunity on multi-SKU production lines and is frequently invisible in aggregate OEE figures.

**Speed loss trend.** Performance rate by line and product over time. Sites that focus only on downtime often miss chronic underperformance at reduced speed that can represent 5 to 15 percent of total capacity loss.

**Utility intensity correlation.** On sites with historian-connected energy metering, correlating OEE with utility consumption per unit produced reveals efficiency differences between products, shifts, and operating modes.

## Choosing an OEE software platform

Several platform approaches are in use at Australian manufacturing sites.

**SCADA-integrated OEE** is available in Ignition via the Sepasoft MES modules and in FactoryTalk Analytics. This approach works well when the site already runs a supported SCADA platform and the OEE requirements are moderate. It avoids introducing a separate system but depends on SCADA infrastructure being well-maintained.

**Standalone MES and OEE platforms** such as Tulip, Critical Manufacturing, or SAP ME/MII offer more extensive MES functionality beyond OEE. These are appropriate for sites with broader MES requirements including electronic work instructions, quality management, and ERP integration.

**Custom BI-based solutions** using Power BI, Tableau, or similar tools connected to SQL historians are commonly built in Australian plants where engineering teams want flexibility and have strong internal SQL and BI capability. These can be highly effective but require ongoing engineering effort to maintain.

**OEE on edge devices** using hardware such as Moxa, Red Lion, or similar industrial gateways is suitable for sites with limited SCADA infrastructure that need production visibility without a full SCADA replacement.

## Steps to a reliable OEE implementation

1. Define planned production time consistently across all lines and shifts before capturing any data.
2. Design reason code structures with production and maintenance teams before implementing any data entry tools.
3. Establish a four-to-eight-week baseline period with manual or semi-automated data collection before investing in automated infrastructure.
4. Automate collection from PLC and historian sources for availability and performance components. Focus operator input on categories that require human context.
5. Build daily and weekly reports before advanced analytics. Simple, reliable reports used in daily meetings create more value than complex dashboards used occasionally.
6. Review data quality monthly. Audit reason code application, check for uncoded downtime, and verify baseline assumptions remain valid.
7. Tie OEE metrics to specific improvement projects with clear success criteria. OEE improvement programs that lack targeted projects produce data but no production gains.

OEE monitoring on Australian production lines delivers the most value when it is simple enough to be used reliably by shift teams, specific enough to direct maintenance and improvement effort, and maintained consistently over time. The technology required is straightforward. The organisational discipline to use data every day is where most programs succeed or fail.
