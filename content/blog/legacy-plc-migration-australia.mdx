Many Australian manufacturing sites are running PLC systems that were installed in the 1990s or early 2000s. PLC-5 and SLC 500 platforms from Allen-Bradley, S7-300 and S7-400 systems from Siemens, and early Modicon platforms are still active on production-critical lines in food and beverage, dairy, packaging, and building products facilities across the country.

These systems still run, but their support position has changed fundamentally. Spare parts are scarce, software support has ended, and the engineers who originally built the systems have often left the business. The risk is not theoretical. It is a failed controller on a Saturday night with no spare, or a SCADA server that cannot be restored because the backup strategy has never been tested.

This guide covers when to migrate, how to plan a staged upgrade on a live site, and what makes the difference between a controlled migration and a crisis.

## Signs that a PLC migration is overdue

Most migration decisions come too late rather than too early. These indicators suggest migration should be planned within the next budget cycle.

**Spare parts are on backorder or unavailable.** If replacement CPUs or I/O modules require four to eight weeks of lead time, a single failure can mean weeks of lost production. This is the most common trigger for emergency migration projects.

**Vendor support has ended.** Allen-Bradley discontinued support for PLC-5 family products in 2022. Siemens S7-300 is in extended support with a 2026 end-of-life deadline. Firmware patches, security updates, and phone support are no longer available. Running unsupported platforms on critical lines is an increasing insurance and compliance risk.

**The software environment is fragile.** If the programming software only runs on one ageing laptop, or the commissioning PC cannot be replaced without losing the ability to connect, the system is more vulnerable than it looks.

**Maintenance knowledge is concentrated in one person.** When the engineer who built the system is the only person who understands it, retirements and staff changes create a significant unplanned risk event.

**Performance and visibility have become limiting.** Legacy systems often lack historian connectivity, alarm reporting, and the data outputs needed for modern OEE and production analysis.

## Migration options: full replacement versus partial modernisation

Not every legacy PLC situation requires a full hardware replacement. Understanding the options helps calibrate scope and capital budget more accurately.

| Approach | What it covers | Best suited for |
| --- | --- | --- |
| Full platform migration | New CPU, I/O, networking, and software | End-of-life hardware, major capability gaps |
| Processor swap with retained I/O | New CPU with adapter modules for existing I/O | Limited capital, hardware still healthy |
| SCADA-only migration | New supervisory layer on existing PLCs | SCADA end-of-life without PLC failure risk yet |
| Staged line-by-line migration | Migrate one production line at a time | Large sites with multiple independent lines |
| Maintenance support only | Remote diagnostics, documentation, spare parts strategy | Systems not yet at end-of-life but poorly supported |

For PLC-5 to ControlLogix migrations, processor swap adapters allow existing 1771 I/O chassis to be reused. This reduces wiring cost and downtime but adds complexity and is only viable if the existing I/O chassis infrastructure is healthy.

## How to plan a migration on a live site

The most reliable migrations treat engineering rigour and production continuity as equal priorities. The following phases apply to most brownfield PLC migration projects.

### Phase 1: Discovery and current-state documentation

Before any design work begins, verify what is actually installed. Existing drawings are frequently wrong. I/O schedules may be incomplete. Network topology is often undocumented. This phase should produce a verified asset list, accurate I/O count, network diagram, and list of integrations to third-party equipment.

This discovery phase also identifies dependencies that affect migration sequence. On a dairy site, for example, CIP logic may interact with process PLCs, utilities PLCs, and third-party chemical dosing systems. These dependencies must be mapped before cutover planning begins.

### Phase 2: Control philosophy and design review

Before rewriting software, review the existing control logic for accuracy and intent. Legacy code often contains undocumented workarounds, bypasses installed during production emergencies, and logic that reflects how the machine actually runs rather than how it was designed to run.

A design review meeting with operations, maintenance, and quality representatives turns the migration into an improvement, not just a like-for-like replacement. This is the best time to rationalise alarms, clean up interlock logic, and improve operator interface design.

### Phase 3: Hardware procurement and FAT preparation

Specify new hardware based on the reviewed design, not the existing BOM. Order spares for new components from day one. Begin building FAT scripts in parallel with code development so testing preparation does not extend the project schedule.

### Phase 4: FAT on new system in workshop

Factory acceptance testing should prove that the new system matches the reviewed control philosophy before it arrives on site. This includes I/O simulation, sequence verification, alarm and interlock checks, and HMI navigation testing.

On migrations, FAT is also the time to verify that integration interfaces with retained equipment work correctly. Simulating upstream and downstream equipment behaviour in the workshop saves significant time during site commissioning.

### Phase 5: Cutover execution

The cutover window is when the old system is disconnected and the new system takes over. For most Melbourne plants, this is a planned shutdown, and every hour counts.

The cutover run sheet should define:

- Pre-start checks and hold points
- The sequence of panel changeovers and termination transfers
- Initial power-up and I/O checkout sequence
- Acceptance criteria before process restart
- Rollback decision point and criteria if issues arise
- Escalation contacts and response rules during the window

Teams that enter a cutover without a detailed run sheet consistently experience extended downtime and preventable incidents.

### Phase 6: Post-commissioning stabilisation

The first two to four weeks after a migration are high-value engineering time. PID loop retuning, alarm setpoint refinement, and operator workflow improvements should all be completed during this period while engineering support is still available and engaged.

## PLC-5 to ControlLogix migration: specific considerations

PLC-5 to ControlLogix is one of the most common migration types in Australian manufacturing. Several technical factors make this migration more involved than it first appears.

**Addressing is fundamentally different.** PLC-5 uses a file and element addressing structure that does not map directly to the tag-based memory model in ControlLogix. Code cannot be converted automatically. Each routine must be reviewed and rewritten.

**Communications change significantly.** PLC-5 systems commonly use DH+ and Remote I/O networks that are not supported in ControlLogix without bridging hardware. Defining what communications infrastructure to retain and what to replace is a key early-stage decision.

**Motion control may require hardware changes.** Servo drives connected via SERCOS or proprietary encoder interfaces may need drive replacements to work with the new platform.

**1771 I/O adapter modules are a useful transition tool.** If the 1771 chassis is in good condition, FlexAdapter or similar modules allow the existing I/O to connect to a ControlLogix processor. This reduces wiring disruption but adds hardware in the signal path.

## S7-300 to S7-1500 migration: specific considerations

Siemens TIA Portal supports migration from S7-300 projects, but this is not a push-button process. Block structure, data type handling, and communications configuration all need review.

**Organisation block structure changes.** The main scan cycle OB and interrupt handling are handled differently in S7-1500. Code that relied on specific S7-300 OB behaviour needs to be reviewed and adapted.

**Symbol table to tag-based model.** S7-300 uses symbolic addressing linked to an absolute address. S7-1500 uses a fully tag-based model. This mapping needs to be validated for each variable.

**Profibus to Profinet migration.** Many S7-300 installations use Profibus for remote I/O and drives. Where Profibus devices are being retained, gateways are required. Where they are being replaced, this is an opportunity to standardise on Profinet.

## Risks that cause migration projects to fail

The following issues cause most migration projects to exceed budget or extend beyond their planned downtime window.

- Incomplete discovery leading to scope gaps discovered during commissioning
- FAT compressed to meet schedule, leaving faults to surface on site
- Undocumented interfaces to third-party equipment identified late in the project
- Cutover planning treated as a project management task rather than an engineering task
- Post-commissioning support ending before the system has fully stabilised
- Spare parts strategy not addressed, leaving the new platform vulnerable from day one

Each of these risks is manageable if identified early. The pattern of avoidable migration failures is almost always traceable to compressed front-end engineering and deferred risk management.

## Practical checklist before starting a PLC migration

1. Current-state documentation verified and gaps identified
2. Control philosophy reviewed and improvement requirements captured
3. Platform selected based on lifecycle, spares, and support, not only hardware cost
4. I/O count and communications interfaces confirmed from physical inspection
5. FAT scope and test scripts drafted before development begins
6. Cutover run sheet with rollback criteria and decision authority defined
7. Spare parts strategy for new platform confirmed before procurement
8. Post-commissioning support and tuning window budgeted
9. Operator and maintenance training scoped and scheduled

A legacy PLC migration in Australia is one of the highest-value engineering investments a manufacturing site can make, but only if it is planned with the same discipline applied to any complex production project. Teams that invest in thorough discovery, honest design review, and structured testing consistently achieve clean cutovers and stable operation within their planned outage windows.
